Lesson 0012: Next.js Data Fetching

Kumuha ng data mula sa iyong Laravel API at i-render ito gamit ang SSG, SSR, at ISR.

Lesson Goal Matutunan ang tatlong data fetching strategies ng Next.js — Static Generation (SSG), Server-Side Rendering (SSR), at Incremental Static Regeneration (ISR) — at kung kailan gagamitin ang bawat isa.
Tech Stack Itinuturo ng lesson na ito ang Next.js 15.x gamit ang Pages Router. Tiyaking ito ang bersyon na ginamit mo sa Lesson 0011. Kung magkaiba ang iyong version, maaaring mag-iba ang mga API na ginagamit dito.

Part 1: Overview — Tatlong Paraan ng Data Fetching

MethodKailan nagfe-fetch?Gamitin kapag...
getStaticProps (SSG) Sa build time Ang data ay bihirang magbago (blog posts, about page)
getServerSideProps (SSR) Sa bawat request Ang data ay laging bago (user dashboard, real-time data)
getStaticProps + revalidate (ISR) Sa build time + periodic revalidation Ang data ay nagbabago pero hindi gaano kadalas (CMS content)
Rule of thumb: SSG > ISR > SSR. Static generation ang pinakamabilis dahil pre-built HTML ang serve. SSR ang pinakamabagal dahil sa bawat request ay nagre-render ang server. Gamitin ang SSR lamang kung kailangan talaga ng real-time data.

Part 2: Static Generation (getStaticProps)

I-connect natin ang Next.js portfolio natin sa Laravel API na ginawa natin sa Phase 2.

Una, siguraduhing naka-run ang iyong Laravel backend:

cd ~/my-blog-api
php artisan serve --port=8000

Ngayon, i-fetch ang blog posts mula sa API gamit ang getStaticProps:

// pages/blog/index.js
import Link from 'next/link';

export default function Blog({ posts }) {
  return (
    <div>
      <h1>Blog</h1>
      {posts.length === 0 ? (
        <p>No posts yet.</p>
      ) : (
        <ul>
          {posts.map(post => (
            <li key={post.id}>
              <Link href={`/blog/${post.id}`}>
                {post.title}
              </Link>
              <p>{post.content.substring(0, 100)}...</p>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}

// This runs at BUILD TIME
export async function getStaticProps() {
  const res = await fetch('http://localhost:8000/api/posts');
  const json = await res.json();

  return {
    props: {
      posts: json.data || [],
    },
  };
}
Note on API URL Dahil ang getStaticProps ay tumatakbo sa server (sa build time), ang localhost:8000 ay ang Laravel server. Sa production, papalitan mo ito ng production URL ng iyong API.

Dynamic Routes

Para sa individual blog post page, gumamit ng dynamic route kasama ang getStaticPaths:

// pages/blog/[id].js

export default function Post({ post }) {
  return (
    <div>
      <Link href="/blog">&larr; Back to Blog</Link>
      <h1>{post.title}</h1>
      <p>By {post.author?.name || 'Unknown'}</p>
      <div>{post.content}</div>
    </div>
  );
}

// Tinutukoy kung anong dynamic paths ang ie-generate
export async function getStaticPaths() {
  const res = await fetch('http://localhost:8000/api/posts');
  const json = await res.json();
  const posts = json.data || [];

  const paths = posts.map(post => ({
    params: { id: String(post.id) },
  }));

  return { paths, fallback: 'blocking' };
}

// Nagfe-fetch ng data para sa specific post
export async function getStaticProps({ params }) {
  const res = await fetch(`http://localhost:8000/api/posts/${params.id}`);
  const json = await res.json();

  return {
    props: {
      post: json.data || json,
    },
  };
}
What is fallback?
  • fallback: false — 404 kung hindi kasama sa paths
  • fallback: true — mag-display ng loading state, then i-generate ang page sa browser
  • fallback: 'blocking' — maghintay hanggang ma-generate ang page sa server (recommended)

Part 3: Incremental Static Regeneration (ISR)

Paano kung may bagong post sa Laravel API? Hindi na kailangan i-rebuild ang buong site — gamitin ang ISR sa pamamagitan ng revalidate:

export async function getStaticProps() {
  const res = await fetch('http://localhost:8000/api/posts');
  const json = await res.json();

  return {
    props: {
      posts: json.data || [],
    },
    // Re-generate ang page every 60 seconds kung may request
    revalidate: 60,
  };
}

Ang revalidate: 60 ay nangangahulugang: "i-serve ang naka-cache na page, ngunit i-revalidate ito sa background every 60 seconds". Ang susunod na bumisita pagkatapos ng 60s ay makakakita ng updated content.

Part 4: Server-Side Rendering (getServerSideProps)

Gamitin ang SSR kung kailangan ng real-time data na unique sa bawat request:

// pages/profile.js — nangangailangan ng fresh data bawat request
export default function Profile({ user }) {
  return (
    <div>
      <h1>{user.name}'s Profile</h1>
      <p>Email: {user.email}</p>
    </div>
  );
}

export async function getServerSideProps() {
  const res = await fetch('http://localhost:8000/api/user', {
    headers: {
      // Sa totoong app, kukuha ka ng token mula sa cookies
      Authorization: 'Bearer ' + process.env.API_TOKEN,
    },
  });
  const json = await res.json();

  return {
    props: { user: json },
  };
}

Part 5: Client-Side Fetching (SWR)

Para sa data na nagbabago sa browser (e.g., pagkatapos mag-submit ng form), gamitin ang client-side fetching gamit ang swr library:

npm install swr
// components/PostList.js
import useSWR from 'swr';

const fetcher = (url) => fetch(url).then(r => r.json());

export default function PostList() {
  const { data, error, isLoading } = useSWR(
    'http://localhost:8000/api/posts',
    fetcher
  );

  if (isLoading) return <p>Loading...</p>;
  if (error) return <p>Error loading posts.</p>;

  return (
    <ul>
      {data?.data?.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
SWR Benefits
  • Automatic re-fetching kapag na-focus ulit ang tab
  • Caching — hindi na magre-request kung may cached data pa
  • Optimistic UI — pwedeng mag-update agad ng UI bago pa dumating ang response
  • Pagination at infinite loading — built-in

Summary

  • getStaticProps — SSG, sa build time, pinakamabilis
  • getStaticPaths — para sa dynamic routes na may SSG
  • revalidate — ISR, periodic background revalidation
  • getServerSideProps — SSR, sa bawat request, real-time
  • Client-side fetching (SWR) — para sa interactive data sa browser
